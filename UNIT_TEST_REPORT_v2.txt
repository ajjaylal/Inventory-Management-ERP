[REGR]  DB-09: profiles RLS policy now causes INFINITE RECURSION
        Old Status: FAIL (missing WITH CHECK)
        New Status: CRITICAL REGRESSION — NEW BUG INTRODUCED
        Root Cause: The fix applied to "Allow admin full access to profiles"
                    changed both USING and WITH CHECK to:
                      EXISTS (SELECT 1 FROM profiles p WHERE p.id = auth.uid()
                              AND p.role = 'Admin')
                    This query reads from the `profiles` table while the policy 
                    IS on the `profiles` table. PostgreSQL evaluates the policy 
                    before allowing the SELECT, which calls the policy again → 
                    infinite recursion.
        Observed Error: `UOM fetch failed: { code: '42P17', details: null, 
                         hint: null, message: 'infinite recursion detected in 
                         policy for relation "profiles"' }`
        Impact: CRITICAL — Every Supabase query that implicitly reads `profiles` 
                now fails with a 500-level error. This blocks:
                  - ALL stock item operations
                  - ALL product operations  
                  - Settings page profile load
                  - Any dashboard query that joins profiles
        How to fix:
          The fix must use `auth.jwt()` or a subquery on `auth.users` instead 
          of re-querying `profiles`. The correct non-recursive pattern is:

          Option A — Use JWT claims (BEST for admins):
            Run in Supabase SQL Editor:

            DROP POLICY IF EXISTS "Allow admin full access to profiles" ON public.profiles;
            CREATE POLICY "Allow admin full access to profiles"
              ON public.profiles FOR ALL
              USING (auth.uid() IS NOT NULL)
              WITH CHECK (auth.uid() IS NOT NULL);

            NOTE: This grants full access to all authenticated users.
            For admin-only enforcement, use Option B.

          Option B — Use auth.users metadata (NO recursion):
            DROP POLICY IF EXISTS "Allow admin full access to profiles" ON public.profiles;
            CREATE POLICY "Allow admin full access to profiles"
              ON public.profiles FOR ALL
              USING (auth.uid() IS NOT NULL)
              WITH CHECK (auth.uid() IS NOT NULL);

            Then enforce admin role at the application layer (check 
            profiles.role === 'Admin' in the RPC functions which are 
            SECURITY DEFINER and bypass RLS anyway).

          Option C — Use a SECURITY DEFINER function to check role:
            CREATE OR REPLACE FUNCTION is_admin()
            RETURNS boolean
            LANGUAGE sql
            SECURITY DEFINER
            SET search_path = public
            AS $$
              SELECT EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() 
                             AND role = 'Admin');
            $$;

            DROP POLICY IF EXISTS "Allow admin full access to profiles" ON public.profiles;
            CREATE POLICY "Allow admin full access to profiles"
              ON public.profiles FOR ALL
              USING (is_admin())
              WITH CHECK (is_admin());

            (The SECURITY DEFINER function bypasses the calling policy,
             breaking the recursion.)

          RECOMMENDED: Use Option A or Option C.
          Apply the SQL immediately — this bug blocks the entire application.

[NEW]   N-01: proxy.ts export name mismatch with Next.js expectation
        Location: /src/proxy.ts
        Issue: File is renamed to proxy.ts and export is `proxy`, but Next.js 16
               actually still expects the exported function to be named 
               `middleware` inside the file. The file is named proxy.ts to 
               resolve the filesystem naming deprecation, but the internal 
               export must remain `middleware`.
        Evidence: Build output shows "ƒ Proxy (Middleware)" — this is correct.
                  However, server logs show proxy.ts processing time (271ms 
                  average), which suggests the route matching is working.
                  Needs to be monitored — if auth protection stops working,
                  this rename could be the cause.
        Impact: LOW — Current behavior appears correct but may break auth 
                middleware routing in edge cases.
        How to fix:
          In src/proxy.ts, change the function name back to `middleware`:

          // Change:
          export async function proxy(request: NextRequest) {

          // To:
          export async function middleware(request: NextRequest) {

          The FILE name is proxy.ts (correct for Next.js 16), but the EXPORT
          name inside the file must still be `middleware`.

[NEW]   N-02: `supabase` client unstable reference in RecordSaleModal.tsx
        Location: /src/components/sales/RecordSaleModal.tsx, Line 82
        Issue: `useEffect(..., [sale, supabase])` — `supabase` is created via
               `createClient()` inside the component body. Because `createClient()`
               returns a new object reference on every render, it will be a
               new value in the dep array every render, causing the useEffect
               to re-run and re-fetch products on every render cycle.
        Impact: MEDIUM — Duplicate API calls on every render. Could cause 
                race conditions if the component re-renders frequently.
        How to fix:
          In RecordSaleModal.tsx line 82, change:
            }, [sale, supabase]);
          To:
            // eslint-disable-next-line react-hooks/exhaustive-deps
            }, [sale]);

[NEW]   N-03: `supabase` client unstable reference in settings/page.tsx
        Location: /src/app/(dashboard)/settings/page.tsx, Line 41

               client is unstable, causing the profile load to re-trigger
